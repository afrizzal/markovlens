# Data Storage — DuckDB + Parquet

## Database

**All persistent data lives in DuckDB** at `data/markovlens.duckdb`.

- File-based, embedded — no server required
- Optimized for analytical queries (column store, vectorized execution)
- Reads Parquet and CSV directly via `SELECT * FROM 'path.parquet'`
- ACID-compliant for transactions
- Schema source of truth: `core/db/schema.sql`

## Connection

Single connection pattern — open once, share via `core/db/connection.py`:

```python
import duckdb
from core.config import settings

_connection: duckdb.DuckDBPyConnection | None = None

def get_connection() -> duckdb.DuckDBPyConnection:
    global _connection
    if _connection is None:
        _connection = duckdb.connect(str(settings.duckdb_path))
        _connection.execute("PRAGMA enable_progress_bar=true;")
    return _connection
```

In Streamlit, wrap with `@st.cache_resource`:

```python
@st.cache_resource
def get_db():
    from core.db.connection import get_connection
    return get_connection()
```

## Query Patterns

### ✅ Use parameterized queries always

```python
result = conn.execute(
    "SELECT * FROM transitions WHERE dataset_id = ? AND period >= ?",
    [dataset_id, min_period],
).df()
```

### ✅ Convert to Pandas at the boundary

```python
df = conn.execute("SELECT ...").df()           # → pd.DataFrame
matrix = conn.execute("SELECT ...").fetchnumpy()  # → dict[str, np.ndarray]
```

### ✅ Use DuckDB's Parquet superpowers

```python
# Read CSV directly, no import step
df = conn.execute("SELECT * FROM 'data/raw/churn.csv' WHERE state = 'active'").df()

# Aggregate without loading whole file into memory
counts = conn.execute("""
    SELECT from_state, to_state, COUNT(*) as n
    FROM 'data/raw/transitions.parquet'
    GROUP BY from_state, to_state
""").df()
```

### ❌ Don't do this

```python
# ❌ String interpolation = SQL injection vector
conn.execute(f"SELECT * FROM transitions WHERE dataset_id = '{user_input}'")

# ❌ Fetching huge result then filtering in Python
df = conn.execute("SELECT * FROM transitions").df()
df = df[df["dataset_id"] == dataset_id]  # too late, already loaded

# ❌ Raw SQL scattered across app/ files
# Wrap queries in core/db/queries.py instead
```

## Schema Migrations

No Alembic. Use simple `CREATE TABLE IF NOT EXISTS` in `core/db/schema.sql`. Run on app startup:

```python
def init_schema(conn: duckdb.DuckDBPyConnection) -> None:
    schema_path = Path(__file__).parent / "schema.sql"
    conn.execute(schema_path.read_text())
```

For breaking changes (e.g., column rename):
1. Add new column nullable
2. Backfill in a script under `scripts/migrate_*.py`
3. Drop old column in a follow-up release
4. Document in `docs/planning/decisions.md`

## File Layout

```
data/
├── markovlens.duckdb           # Main DB file (gitignored)
├── markovlens.duckdb.wal       # Write-ahead log (gitignored)
├── raw/                        # Original Kaggle/source files (gitignored)
│   ├── ecommerce_brand_share.csv
│   └── telco_churn.csv
├── processed/                  # Cleaned, ready-to-load Parquet (gitignored)
│   └── *.parquet
└── cache/                      # Computed matrices / simulation snapshots (gitignored)
```

`data/` contents are gitignored — datasets are regenerated by `scripts/seed_data.py` from the original Kaggle CSVs.

## Core Tables

See [docs/DATABASE.md](../../docs/DATABASE.md) for the full schema with ERD. Summary:

| Table | Purpose |
|---|---|
| `datasets` | Registered datasets (id, domain, name, source, row count, n_states) |
| `transitions` | Raw transition observations (long format: entity_id, period, from_state, to_state) |
| `transition_matrices` | Cached computed matrices (matrix_json, n_observations) |
| `simulation_runs` | Monte Carlo result cache (final_distribution_json, quantile_paths_json) |
| `forecasts` | Forecast outputs with accuracy metrics |
| `scenarios` | Saved what-if scenarios |

## Backup Strategy

- Local dev: nothing — regenerable from raw CSVs
- Production (Streamlit Cloud): same — datasets loaded on cold start
- For real deployment: consider mounting persistent disk; back up `markovlens.duckdb` daily

## Performance Tips

- DuckDB is 10-100x faster than Pandas for aggregations on > 100k rows
- Use SQL `GROUP BY` instead of `pd.DataFrame.groupby()` for performance-critical paths
- For repeated heavy queries, materialize a view: `CREATE VIEW transition_summary AS ...`
- `EXPLAIN ANALYZE` to debug slow queries

## What NOT to Do

- ❌ Use SQLite instead — slower for our workload
- ❌ Use SQLAlchemy ORM — overkill, DuckDB native API is simpler
- ❌ Store raw datasets in DB — Parquet on disk is better; query directly
- ❌ Commit `*.duckdb` files to git — they grow, they're binary, gitignored
